ProductLoop OS package atlas

ProductLoop OS presents Maqam and eight modular governance packages through one named-namespace dependency.

The product model Start with Maqam for the focused execution boundary. Start with productloop-os when one application needs the broader package family and tested composition helpers.

Install the umbrella

npm install productloop-os@0.2.0

npx productloop-os@0.2.0 demo

productloop-os

public 0.2.0 · npm

Creates a deny-by-default composition with named namespaces, separate ledgers and registries, tested schema adapters, and a dependency doctor.

import { createProductLoopOS } from "productloop-os";

const os = createProductLoopOS({
  policyBundle: {
    schemaVersion: "ajnas.policy.bundle.v1",
    id: "local-policy",
    version: "1.0.0",
    defaultEffect: "deny",
    rules: [{
      id: "allow-echo",
      effect: "allow",
      reason: "Reviewed local tool",
      match: { toolNames: ["local.echo"], risks: ["low"] }
    }]
  }
});

os.tools.register({
  name: "local.echo",
  description: "Return local structured input",
  risk: "low",
  execute: ({ input }) => input
});

const result = await os.runtime.run({
  name: "hello",
  input: { message: "hello" },
  steps: [{
    id: "echo",
    run: (context) => context.callTool("local.echo", context.input)
  }]
});

os.importRuntimeEvents();
console.log(result.status, os.trace.events.length);

Not a model, hosted control plane, identity system, database, distributed scheduler, secret manager, sandbox, browser engine, or crawler fleet.

maqam

public 0.2.3 · 0.2.4 candidate pending approval · npm

Provides the higher-level exact approval gateway, workflow runtime, evidence ledger, worker adapters, local console, and bounded HTTP crawler.

import { PolicyEngine, ToolGateway } from "maqam";

const gateway = new ToolGateway({
  policyEngine: new PolicyEngine({ allowedTools: ["local.echo"] })
});

gateway.registerTool("local.echo", async (input) => input, {
  effects: []
});

console.log(await gateway.call(
  "local.echo",
  { message: "hello" },
  { runId: "local_1" }
));

Current approval, trace, runtime, and evidence state is in-process unless the host exports, authenticates, and protects it.

ajnas-runtime

public 0.2.0 · npm

Runs ordered workflow steps, evaluates every registered tool call, emits hash-linked provenance receipts, and optionally writes file-backed snapshots.

import {
  AgentRuntime,
  InMemoryProvenanceSink,
  ToolRegistry
} from "ajnas-runtime";

const tools = new ToolRegistry().register({
  name: "local.echo",
  description: "Return local input",
  risk: "low",
  execute: async ({ input }) => input
});

const runtime = new AgentRuntime({
  tools,
  provenance: new InMemoryProvenanceSink(),
  policy: {
    evaluate: async () => ({ decision: "allow", reason: "local fixture" })
  }
});

console.log(await runtime.run({
  name: "echo",
  input: { message: "hello" },
  steps: [{
    id: "echo",
    run: (context) => context.callTool("local.echo", context.input)
  }]
}));

It is an execution kernel, not a provider, model loop, remote queue, or trusted reviewer identity service.

ajnas-policy

public 0.1.1 · npm

Evaluates versioned bundles into allow, deny, or require_approval decisions and records replayable audit receipts.

import {
  PolicyDecisionLedger,
  createDeclarativePolicyEngine
} from "ajnas-policy";

const bundle = {
  schemaVersion: "ajnas.policy.bundle.v1",
  id: "release-policy",
  version: "1.0.0",
  defaultEffect: "deny",
  rules: [{
    id: "review-publish",
    effect: "require_approval",
    reason: "Owner review required",
    match: { toolNames: ["packages.publish"] }
  }]
};

const audit = new PolicyDecisionLedger();
const policy = createDeclarativePolicyEngine({ bundle, audit });
const decision = await policy.evaluate({
  runId: "run_1",
  stepId: "publish",
  tool: { name: "packages.publish", risk: "critical" },
  input: { destination: "npm" },
  metadata: { purpose: "release" }
});

console.log(decision, audit.verify());

Policy code receives metadata, not the execution callback. A policy decision has value only when the real execution path enforces it.

ajnas-approvals

public 0.1.1 · npm

Models staged review, quorum, delegation, expiry, escalation, decisions, and hash-linked audit history without choosing a notification transport.

import {
  ApprovalQueue,
  createReleaseApprovalSubject
} from "ajnas-approvals";

const workflow = {
  schemaVersion: "ajnas.approval.workflow.v1",
  id: "release-approval",
  version: "1.0.0",
  name: "Release approval",
  owner: "release-governance",
  stages: [{
    id: "owner-review",
    name: "Owner review",
    reviewers: [{ id: "release-owner", kind: "role" }],
    quorum: { strategy: "any" }
  }],
  policy: { preventRequesterSelfApproval: true, defaultTtlHours: 48 }
};

const queue = new ApprovalQueue();
const ticket = queue.request({
  workflow,
  subject: createReleaseApprovalSubject({
    packageName: "example-package",
    version: "1.0.0",
    changeSummary: "Reviewed release"
  }),
  requestedBy: "release-bot",
  reason: "External publish requires approval"
});

console.log(ticket.status);

The host must authenticate reviewers, persist tickets, send notifications, and protect restored approval state from tampering.

ajnas-provenance

public 0.1.2 · npm

Records append-only events with SHA-256 receipt chains, redacts common secret fields, and creates verifiable or signed trace bundles.

import {
  TraceLedger,
  createTraceBundle,
  verifyTraceBundle
} from "ajnas-provenance";

const ledger = new TraceLedger({ traceId: "trace_release_1" });
ledger.record({
  actor: "release-bot",
  action: "approval.requested",
  subject: { type: "package", id: "demo@1.0.0" },
  source: { system: "release", id: "gate" },
  data: { destination: "npm" }
});

const bundle = createTraceBundle(ledger.events, {
  generatedBy: "release-bot",
  purpose: "release-gate"
});

console.log(verifyTraceBundle(bundle));

Receipts make later alteration detectable. They do not make storage immutable or establish that a recorded claim is true.

ajnas-skills-registry

public 0.2.0 · npm

Validates and registers signed skill manifests with capability, permission, publisher, compatibility, and license metadata.

import { generateKeyPairSync } from "node:crypto";
import {
  SkillRegistry,
  signSkillManifest
} from "ajnas-skills-registry";

const manifest = {
  schemaVersion: "ajnas.skill.v1",
  id: "com.example.local-summary",
  name: "Local Summary",
  version: "1.0.0",
  description: "Summarize reviewed local notes.",
  license: "MIT",
  entrypoint: { type: "module", value: "./dist/index.js" },
  compatibility: { ajnasRuntime: "^0.2.0" },
  capabilities: [{
    name: "notes.summarize",
    description: "Summarize reviewed notes.",
    risk: "medium"
  }],
  permissions: {
    network: "none",
    filesystem: "read",
    externalPublish: false,
    humanApproval: "on_high_risk"
  },
  trust: {
    source: "first-party",
    publisher: "example",
    repository: "https://example.com/repository",
    licenseEvidence: {
      kind: "repository",
      checkedAt: "2026-07-16T00:00:00.000Z"
    }
  }
};

const { privateKey, publicKey } = generateKeyPairSync("ed25519");
const signed = signSkillManifest(manifest, {
  algorithm: "ed25519",
  keyId: "local-key",
  privateKey
});

const registry = new SkillRegistry({
  keyResolver: (keyId) => keyId === "local-key" ? publicKey : null
});
registry.register(signed, { actor: "owner", source: "local-review" });

A valid signature proves key possession for the signed digest. It does not audit, sandbox, install, or make executable skill code safe.

ajnas-connectors

public 0.1.1 · npm

Validates connector manifests and turns declared permissions, operations, data classes, publisher metadata, and review state into trust decisions.

import {
  ConnectorAuditLedger,
  ConnectorRegistry,
  evaluateConnectorTrust,
  validateConnectorManifest
} from "ajnas-connectors";

const validation = validateConnectorManifest(manifest);
if (!validation.valid) {
  throw new Error(JSON.stringify(validation.issues));
}

const audit = new ConnectorAuditLedger();
const registry = new ConnectorRegistry({ audit });
registry.register(manifest, { actorId: "platform-engineer" });

const trust = evaluateConnectorTrust(manifest, { request: invocation });
console.log(trust.decision, audit.verify());

This snippet expects your reviewed manifest and invocation objects. The package does not create clients or call external services.

ajnas-evals

public 0.1.1 · npm

Evaluates normalized run artifacts against explicit status, JSON, event, policy, approval, provenance, connector, and browser-step assertions.

import {
  runEvalSuite,
  validateEvalSuite,
  verifyEvalReport
} from "ajnas-evals";

const validation = validateEvalSuite(suite);
if (!validation.valid) {
  throw new Error(JSON.stringify(validation.issues));
}

const report = await runEvalSuite(
  suite,
  async (testCase) => runArtifacts[testCase.id],
  { generatedBy: "release-gate" }
);

console.log(report.summary);
console.log(verifyEvalReport(report));

This snippet expects a reviewed suite and normalized runArtifacts. The package does not run a provider or external service by itself.

ajnas-browser-research

public 0.1.2 · npm

Runs declarative research plans around replaceable browser, search, crawl, and note adapters with policy decisions, citations, replay, and provenance.

import {
  ReplayBrowserAdapter,
  runBrowserResearchPlan,
  verifyBrowserResearchReport
} from "ajnas-browser-research";

const report = await runBrowserResearchPlan(plan, {
  adapter: new ReplayBrowserAdapter(replayRecords),
  runId: "run_release_fixture"
});

console.log(report.status);
console.log(verifyBrowserResearchReport(report).valid);

This snippet expects a plan and replay records. Production browser engines, credentials, sandboxes, network policy, tenant isolation, and access-control compliance remain deployment responsibilities.

Composition rules

  1. Keep package stores, registries, ledgers, and transactions distinct unless the host defines a real synchronization contract.
  2. Use correlation IDs, idempotency keys, an outbox or retry policy, and reconciliation for cross-package workflows.
  3. Do not treat a manifest, policy decision, approval ticket, or trace receipt as an operating-system sandbox.
  4. Route the real side effect through Maqam when exact run, tool, and input binding is required.
