Governed research sources

Maqam 0.3.0 adds a provider-neutral source registry: applications can normalize research records, choose ordered backends, run bounded host-defined checks, and route reads through an existing governed tool boundary.

The model

ResearchDocument
A frozen versioned record with HTTP or HTTPS URI, text or Markdown, source adapter identity, channel, retrieval time, authors, metadata, and citations.
Source adapter
A static id, channel, tool name, priority, declared capabilities, optional direct reader, and optional host-defined check.
Registry
Deterministic ordering, explicit backend preferences, normalized output, attempt records, and strict error classification.
ToolCaller
The host-owned call function used by route(). Bind it to a configured ToolGateway to enforce Maqam policy.
The registry is not the enforcement boundary ResearchSourceRegistry.route() is governed only when its ToolCaller invokes a real configured gateway or equivalent host control. routeUngoverned() bypasses that gateway, deliberately calls adapter code directly, and is labelled ungoverned in its result.

Register and route one local source

This fixture has no network or model call. It shows where an application connects the registry to Maqam's gateway.

import {
  PolicyEngine,
  ResearchSourceRegistry,
  ToolGateway,
  defineResearchSourceAdapter
} from "maqam";

const adapter = defineResearchSourceAdapter({
  id: "docs.local",
  channel: "docs",
  toolName: "research.docs.local",
  priority: 10,
  authentication: "none",
  capabilities: ["read"],
  read: async ({ uri }) => [{
    uri,
    title: "Reviewed local fixture",
    text: "A deterministic source record."
  }],
  check: async () => ({
    status: "ready",
    message: "Local fixture is available."
  })
});

const gateway = new ToolGateway({
  policyEngine: new PolicyEngine({
    allowedTools: [adapter.toolName]
  })
});
gateway.registerTool(adapter.toolName, adapter.read, {
  effects: []
});

const registry = new ResearchSourceRegistry({
  adapters: [adapter],
  toolCaller: { call: gateway.call.bind(gateway) }
});

const result = await registry.route({
  channel: "docs",
  input: { uri: "https://example.com/reviewed" }
}, {
  runId: "research_1"
});

console.log(result.governance.mode);
console.log(result.documents[0]);

Fallback is narrow and fail closed

Adapters are ordered by explicit backend preference, then priority and registration order. The registry can try the next backend only after an explicit ResearchSourceUnavailableError. Unknown exceptions, malformed output, HTTP authorization or server failures, policy denial, approval requirements, security errors, and authentication requirements stop the route.

  • Authenticated adapters require allowAuthenticated: true; the host still owns credentials and session policy.
  • Adapter output is fully validated before any normalized document collection is returned.
  • Direct shell commands, cookie extraction, package installation, and implicit login are not adapter configuration fields.
  • A backend preference changes ordering only. It cannot add an adapter or weaken its tool policy.

Source doctor

registry.doctor() runs each registered in-process check independently with a timeout and returns ready, degraded, unavailable, blocked, or error status. One broken check cannot hide the others. The timeout aborts the supplied signal, but host code must cooperate with cancellation.

const report = await registry.doctor({
  channel: "docs",
  timeoutMs: 2_000
});

console.log(report.status);
console.table(report.checks.map((check) => ({
  adapter: check.adapter.id,
  status: check.status,
  message: check.message
})));
What doctor proves Only the adapter's registered host-defined check. Built-in checks are local, but Maqam cannot prove a custom check is offline or side-effect free. Doctor does not prove live credentials, provider availability, content truth, or production capacity unless a separately reviewed check explicitly tests those things.

RSS, Atom, and feed-aware crawling

createWebCrawlerSourceAdapter(createCrawlerTool({...})) exposes Maqam's bounded crawler under the fixed research.web-crawler.direct source identity. The adapter adds normalized documents and routing metadata; it does not create an alternate fetch, login, cookie, or credential path. DNS, redirect, robots, origin, byte, request, depth, and duration controls still belong to the supplied crawler and gateway context.

The offline parseRssAtom() parser accepts already-retrieved RSS 2.0 or Atom XML. It bounds input and output, rejects DTD and entity declarations, resolves only credential-free HTTP or HTTPS links, sanitizes content, deduplicates entries, and records SHA-256 content hashes.

import { parseRssAtom } from "maqam";

const feed = parseRssAtom(xmlFromGovernedReader, feedUrl, {
  maxItems: 100,
  maxInputBytes: 2 * 1024 * 1024,
  maxTextChars: 20_000
});

console.log(feed.items);
console.log(feed.contentHash);

The parser performs no network access. createRssAtomResearchAdapter() and createRssAtomSourceAdapter() require a host-supplied reader. The built-in crawler can discover linked feeds with includeFeeds: true while preserving its existing DNS, redirect, robots, byte, duration, origin, and request limits.

Feed support is not browser automation Maqam does not execute page JavaScript, extract browser cookies, bypass login or anti-bot controls, rotate proxies, or provide a distributed crawl fleet.

How this relates to Agent Reach

Agent Reach is an MIT-licensed Python capability layer with broader platform-specific channel coverage. It installs, checks, and routes upstream platform tools, and can use local command-line tools, authenticated sessions, or browser cookies where a platform requires them.

Maqam 0.3.0 adopts the useful architectural ideas of ordered backends, explicit health status, and real checks, but applies them to a narrower TypeScript governance contract. It does not copy Agent Reach's auto-installation, cookie extraction, login reuse, platform-specific command wrappers, or channel coverage.

Production boundary checklist

  1. Register each backend under one static adapter id, channel, and tool name.
  2. Bind route() to a reviewed ToolCaller; do not describe direct reads as governed.
  3. Keep provider credentials, cookie stores, command paths, and client construction in trusted host code.
  4. Classify unavailable backends separately from policy, approval, authentication, and security denials.
  5. Validate normalized documents before storing them or making claims from their content.
  6. Use network egress controls, authorization, terms compliance, rate limits, retention, and redaction outside the registry.
