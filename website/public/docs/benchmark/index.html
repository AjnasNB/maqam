Maqam Governance Evaluation Suite

MGES v1 keeps local performance regression evidence separate from deterministic governance-boundary conformance.

Verdict

MGES is project-defined, not globally accepted MGES is not a certification, security score, penetration test, formal proof, competitor benchmark, capacity result, end-to-end agent benchmark, or production SLA.

The useful claim is narrow: the MGES method, environment, source fingerprint, raw observations, project checks, and deterministic fixtures are public and reproducible. Use it as a release regression and conformance profile for Maqam's approval boundary, never as an industry seal.

How MGES fits the benchmark landscape

No cited industry initiative or agent benchmark certifies exact side-effect approval integrity across frameworks. The closest references answer different questions, so they should complement MGES rather than be presented as substitutes or endorsements.

Industry references, their evaluation scope, and relationship to the project-defined MGES suite

Recommended presentation: report MGES beside its environment and raw artifacts, map relevant failure modes to OWASP, follow emerging NIST work, and use capability benchmarks separately when evaluating the model or agent as a whole.

Local-call performance method

The fixture is a sequential asynchronous in-process handler that returns input.value + 1. One path calls it directly. The other path routes it through ToolGateway, an allowlist PolicyEngine, canonical input handling, limits, and tracing.

Recorded parameters for the MGES local-call performance method
ParameterRecorded value
EnvironmentNode 24.15.0, Windows x64, AMD Ryzen 7 4800H
Observations30 per variant, each in a fresh child process
OrderAlternating direct/governed and governed/direct paired rounds
Direct batch250,000 warmup iterations, then 5,000,000 measured iterations
Governed batch500 warmup iterations, then 5,000 measured iterations
Timerprocess.hrtime.bigint(), process startup outside the measured interval
Interval10,000 deterministic percentile-bootstrap resamples for the sample median

Excluded from the timed fixture: model inference, network I/O, filesystem I/O, durable storage, human review time, process startup, and concurrent load.

CPU affinity, processor frequency, and background load were not controlled. Re-run on your deployment class before setting operational thresholds.

Project publication checks

MGES v1.1 criteria version 2 passed all four required checks: at least 30 observations, governed variation at or below 10%, and median timed batches of at least 100 milliseconds for both paths. Direct-path variation remains a published diagnostic because the near-zero baseline is dominated by CPU-frequency noise; it also passed in this run. These are Maqam stability criteria, not external acceptance rules.

Governance conformance profile

Each named fixture tests one failure or integrity property on the recorded source fingerprint.

Policy and input boundary
  1. MGES-C01: deny-by-default construction.
  2. MGES-C02: policy denial blocks dispatch.
  3. MGES-C03: policy evaluation fails closed.
  4. MGES-C04: accessor input is rejected without executing getters, policy, or the handler.
Approval and execution integrity
  1. MGES-C05: approval binds run, tool, and canonical input.
  2. MGES-C06: approval is single-use by default.
  3. MGES-C07: per-run call limit is enforced.
  4. MGES-C08: handler input is detached and deeply immutable.
Evidence, atomicity, and trace
  1. MGES-C09: cross-run evidence does not support a claim.
  2. MGES-C10: multi-approval consumption is atomic.
  3. MGES-C11: gateway evidence attribution is scoped to run, task, and tool.
  4. MGES-C12: denial trace is coded and redacts a token-shaped field.
Governed source routing
  1. MGES-C13: a source policy denial stops every backend before dispatch.
  2. MGES-C14: ordinary source unavailability falls back in order and binds normalized provenance.

Passing confirms only these fixtures. It does not prove the absence of defects or cover an adversarial deployment environment.

Safe publication wording

MGES v1.1.0 local-call profile on Node 24.15.0, Windows x64,
and Ryzen 7 4800H: 123.773 microseconds median per governed
call, with a 122.659 to 126.087 microsecond 95% bootstrap
interval, 30 fresh-process observations, and 2.930% governed CV.
The separate boundary profile passed 14 of 14 deterministic
fixtures. Project-defined local regression evidence, not a
standard, certification, security score, capacity result,
competitor benchmark, or production SLA.

Always keep the environment, profile, interval, observation count, variability, caveat, and raw-artifact link beside the headline number.

Standards context

The broader standards landscape is active. NIST has launched an AI Agent Standards Initiative, but that does not make MGES a NIST benchmark. OWASP supplies a peer-reviewed risk vocabulary, not a pass/fail certification. Node.js benchmark guidance is used as a reproducibility reference. SPEC reporting policy is a useful example of disclosure discipline, but MGES is not a SPEC suite.
