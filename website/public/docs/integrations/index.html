Maqam 0.3.2 integrations

Use static tool adapters for side effects and static research-source adapters for normalized reads. In both cases, the host supplies the real client and keeps the enforcement path explicit.

Published 0.3.0 API line The 0.2 tool-adapter contract remains available. The 0.3 release line adds source adapters, ordered registry routing, normalized research documents, bounded host-defined source checks, RSS or Atom helpers, and guarded structural browser actions. Pin maqam@0.3.2 so examples resolve to one reviewed API surface.

For source review, inspect the exact npm gitHead, check out that commit, and run repository verification:

git clone https://github.com/AjnasNB/maqam.git
cd maqam
# Replace <npm-git-head> with the registry value for 0.3.0
git checkout <npm-git-head>
npm ci
npm run verify
npm pack --dry-run

Verify the public registry version, provenance, integrity, and tagged source, then install the pinned package with npm install maqam@0.3.2.

Boundary statement Maqam does not bundle provider SDKs, a native MCP client or server, provider discovery, protocol authentication, an HTTP service client, a connector marketplace, or secret storage.

Define a static adapter

The adapter name is the policy identity. Declare transport, effects, risk, description, and invocation explicitly.

import {
  PolicyEngine,
  ToolGateway,
  defineToolAdapter,
  registerToolAdapter
} from "maqam";

const adapter = defineToolAdapter({
  name: "function.slug",
  transport: "function",
  description: "Create a URL slug without external I/O.",
  effects: [],
  risk: "low",
  invoke: async ({ value }) => ({
    slug: value.toLowerCase().replaceAll(" ", "-")
  })
});

const gateway = new ToolGateway({
  policyEngine: new PolicyEngine({
    allowedTools: [adapter.name]
  })
});

registerToolAdapter(gateway, adapter);
console.log(await gateway.call(
  adapter.name,
  { value: "Maqam Release" },
  { runId: "release_1" }
));

The invoked adapter receives the detached, deeply frozen input that policy evaluated. Adapter output remains trusted input from the host integration. Defining an adapter does not validate or record that output as evidence.

Research source adapters

defineResearchSourceAdapter() gives one backend a static adapter id, channel, tool name, priority, authentication mode, capabilities, optional direct reader, and optional host-defined check. ResearchSourceRegistry.route() invokes the selected tool name through a required host ToolCaller and normalizes output into frozen ResearchDocument records.

Different adapter, same enforcement ruleRegister the source adapter's read function under its declared tool name and bind the registry ToolCaller to that gateway. Calling adapter.read or routeUngoverned() directly bypasses gateway policy.

The full ordering, fallback, doctor, RSS or Atom, and Agent Reach comparison is in the governed sources guide.

SDK operation

Construct and authenticate the client in trusted host configuration. Do not place API tokens in canonical adapter input or evidence.

const issueAdapter = defineToolAdapter({
  name: "sdk.issue.create",
  transport: "sdk",
  description: "Create an issue through the configured SDK client.",
  effects: ["network:write"],
  risk: "high",
  invoke: (input) => issueSdk.issues.create(input)
});

registerToolAdapter(gateway, issueAdapter);

Configure approvalRequiredEffects: ["network:write"] when this operation must pause for exact human approval.

HTTP operation

Keep the destination in canonical input so policy can inspect it, then compare the effective origin against the gateway-authorized origin scope.

const httpAdapter = defineToolAdapter({
  name: "http.release.enqueue",
  transport: "http",
  description: "Queue a release through a configured HTTP client.",
  effects: ["network:write", "release:queue"],
  risk: "high",
  async invoke(input, context) {
    const origin = new URL(input.endpoint).origin;
    if (!context.authorizedOrigins.includes(origin)) {
      throw new Error(`Origin '${origin}' was not authorized.`);
    }

    return httpClient.request({
      method: "POST",
      url: input.endpoint,
      json: { release: input.release }
    });
  }
});
HTTP is not made safe by a label The host must constrain redirects, DNS resolution, proxies, private networks, response bytes, retries, authentication, rate limits, and deadlines.

MCP-shaped operation

Register one static local adapter for one intended remote tool. Do not let untrusted input choose an arbitrary remote tool name behind one policy identity.

const mcpAdapter = defineToolAdapter({
  name: "mcp.github.create_issue",
  transport: "mcp",
  description: "Call one configured remote tool through the host client.",
  effects: ["network:write"],
  risk: "high",
  invoke: (input) => mcpClient.callTool({
    name: "create_issue",
    arguments: input
  })
});

registerToolAdapter(gateway, mcpAdapter);
Not native MCP support This is an MCP-shaped host wrapper. The host owns connection lifecycle, discovery, schema validation, authentication, cancellation, protocol errors, and remote permissions.

ProductLoop OS composition

Public productloop-os@0.2.3 exposes a Maqam gateway. The adapter helpers shown here are available in maqam@0.3.2. Registration remains explicit. See the ProductLoop OS v0.2.3 release.

import { createProductLoopOS } from "productloop-os";
import {
  defineToolAdapter,
  registerToolAdapter
} from "maqam";

const adapter = defineToolAdapter({
  name: "sdk.issue.create",
  transport: "sdk",
  description: "Create an issue through the host SDK.",
  effects: ["network:write"],
  risk: "high",
  invoke: (input) => issueSdk.issues.create(input)
});

const productLoop = createProductLoopOS({
  maqamPolicy: { allowedTools: [adapter.name] }
});
registerToolAdapter(productLoop.maqamGateway, adapter);

const result = await productLoop.maqamGateway.call(
  adapter.name,
  { title: "Review release" },
  { runId: "productloop_release_1" }
);

console.log(result);

ProductLoop connector manifests, approvals, provenance, runtime events, and Maqam state remain distinct. Define correlation, mapping, idempotency, persistence, retry, and reconciliation explicitly.

Google ADK

Illustrative and not tested in this repository Maqam does not ship a native Google ADK integration. The current ADK TypeScript API can expose a host-defined function tool whose execute handler calls a Maqam gateway.

The official ADK TypeScript quickstart uses FunctionTool with Zod parameters. The following sketch shows the integration seam for a pure local operation. Pin and verify your ADK version before production use.

import { FunctionTool } from "@google/adk";
import { z } from "zod";
import {
  PolicyEngine,
  ToolGateway,
  defineToolAdapter,
  registerToolAdapter
} from "maqam";

const adapter = defineToolAdapter({
  name: "function.slug",
  transport: "function",
  description: "Create a local slug.",
  effects: [],
  risk: "low",
  invoke: ({ value }) => ({
    slug: value.toLowerCase().replaceAll(" ", "-")
  })
});

const gateway = new ToolGateway({
  policyEngine: new PolicyEngine({ allowedTools: [adapter.name] })
});
registerToolAdapter(gateway, adapter);

export const governedSlug = new FunctionTool({
  name: "governed_slug",
  description: "Create a URL slug through the Maqam gateway.",
  parameters: z.object({ value: z.string() }),
  execute: ({ value }) => gateway.call(
    adapter.name,
    { value },
    { runId: crypto.randomUUID() }
  )
});

For a high-risk operation, the host must keep a stable run id, authenticate the reviewer, store the approval safely, and retry the exact same input with the approval id. ADK also has its own action-confirmation feature. Compare its semantics with your requirements instead of layering approvals automatically.

Direct use of ADK MCPToolset bypasses Maqam unless each intended operation is routed through a Maqam adapter and direct paths are removed.

Microsoft Agent 365

Architecture guidance, not a shipped integration Maqam does not ship or test a native Microsoft Agent 365 SDK adapter. Agent 365 supplies enterprise identity, observability, notifications, governance, and Microsoft 365 tool access around agents built with other runtimes.

Wrap individual Agent 365 SDK operations behind static Maqam adapter names when exact local gateway approval is required. Keep Entra identity, blueprint policy, Work IQ permissions, and Microsoft-side governance enabled.

Do not assume that calling addToolServersToAgent governs later tool calls through Maqam. Automatic server registration is outside the Maqam gateway unless each later operation is routed back through an explicit adapter. Work IQ MCP documentation currently describes a preview surface, so verify current tenant availability and contract details.

Run the conformance probe

import { runToolAdapterConformance } from "maqam";

const report = await runToolAdapterConformance(adapter, {
  input: { value: "Fixture Value" },
  verifyOutput: (output) => output.slug === "fixture-value"
});

if (!report.passed) process.exitCode = 1;
console.log(JSON.stringify(report, null, 2));

Run this only against a fixture, emulator, or sandbox. It invokes the adapter once. A pass is not protocol certification, a security score, a distributed-systems test, a production SLA, or proof that bypass paths are closed.

Deployment checklist

  1. Give each operation one static adapter name and declare every meaningful effect.
  2. Configure allowlists and exact approval rules before registration.
  3. Keep credentials in trusted host configuration, outside canonical input and evidence.
  4. Enforce transport timeouts, size limits, rate limits, network controls, retries, and response validation.
  5. Run local conformance against a fixture, then provider-specific integration tests in a restricted environment.
  6. Remove or independently block every direct client path that could bypass the adapter.
  7. Persist and reconcile state when in-process storage is insufficient.

Integration principle

One static identity per real operation.
