Security model

Maqam is an application-level enforcement boundary. It cannot make an unsafe tool, provider, host, or deployment safe by itself.

Route the real side effect An operation is governed only when the application sends the real call through the registered ToolGateway path.

What the boundary covers

Before execution
Explicit policy, allowed tools and origins, effects, risk, tenant limits, and approval requirements.
At approval
Binding to run id, tool name, and canonical input hash, with one-use consumption by default.
At dispatch
Detached and deeply frozen handler input, call ceilings, deadlines, and registered adapter metadata.
After execution
Trace status, coded errors, approval consumption, redacted input fields, and explicitly recorded evidence or claims.

Outside the boundary

  • Unregistered functions, direct SDK clients, direct HTTP calls, direct MCP calls, and provider-internal actions.
  • Operating-system sandboxing, containers, virtual machines, network egress controls, credential brokers, and secret stores.
  • Authentication of reviewer identity or cryptographic authentication of restored approval JSON.
  • Truth of a source or claim, rollback of permitted side effects, and durable restart-safe workflow checkpoints.

Policy controls

  • Configure explicit allowedTools and allowedOrigins. Empty allowlists deny by default.
  • Use unrestricted options only when an intentionally broad policy has been reviewed.
  • Treat handler-declared effects as a minimum set. Registration metadata can add effects but cannot remove handler effects.
  • Set tenant maxToolCalls and maxRuntimeMs. A workflow can lower but cannot raise those ceilings.
  • Fail closed when policy evaluation throws or returns an invalid decision.

Exact approvals

Require approval for write, publish, send, billing, account, credential, destructive, and production-change effects.

import { ApprovalQueue, PolicyEngine, ToolGateway } from "maqam";

const approvals = new ApprovalQueue();
const policy = new PolicyEngine({
  allowedTools: ["release.publish"],
  approvalRequiredEffects: ["publish"],
  maxToolCalls: 1
});

const gateway = new ToolGateway({
  policyEngine: policy,
  approvalQueue: approvals
});

Protect serialized approval queues as trusted security state. Structural validation detects malformed records, but serialization does not prove who approved a record or prevent a trusted-storage attacker from replacing it.

Adapter controls

  • Use static adapter identities. Never hide arbitrary remote tool selection behind one low-risk local name.
  • Keep credentials in trusted host configuration and pass only the fields each client needs.
  • Run conformance against fixtures, then run restricted provider integration tests.
  • Inspect the application for direct client paths that bypass the gateway.
  • Validate external output before using it as evidence or passing it to another side effect.

Governed browser controls

Maqam 0.3.1 registers bounded observe, preview, apply, and submit operations around a host-owned browser driver. Maqam does not supply the browser engine, profile, login, credential store, planner, or operating-system sandbox.

  • Configure exact canonical HTTP(S) origins in both adapter and policy. The driver receives only origins named by that exact request after scope intersection.
  • Keep observe and preview read-only, and revalidate the exact session, page, origin, and revision before dispatch.
  • Block external protocols, downloads, filesystem reads and writes, file pickers, clipboard reads and writes, permission prompts, print dialogs, and modal dialogs in the host before apply or submit.
  • Require the all-false effects result, but treat it as a host attestation checked after dispatch rather than rollback or proof that an untrusted driver is truthful.
  • Repeat preview after process restart and never call the raw browser driver as a policy bypass.

See the governed-browser guide for the structural contract and complete trust boundary.

Research source routing

The 0.3.0 source registry is an ordering and normalization layer. It becomes a governed path only when route() receives a ToolCaller bound to a real configured gateway. A direct adapter read or routeUngoverned() remains outside that enforcement path.

  • Give every source backend one static adapter id, channel, and tool name.
  • Do not fall through after policy, approval, authentication, or crawler security denials.
  • Require explicit allowAuthenticated for adapters that declare authentication, then keep credentials and session state in trusted host code.
  • Treat doctor results as offline adapter checks, not live credential, network, provider, or content-truth verification.
  • Validate every adapter response into a bounded ResearchDocument before storing it or using it as evidence.
No implicit platform accessSource adapter declarations cannot request shell commands, package installation, cookie extraction, or implicit login. A host can still write unsafe adapter code, so review the actual reader and execution environment.

Crawler controls

The built-in crawler accepts public HTTP or HTTPS targets, checks all DNS answers, pins each connection, re-authorizes redirects, rejects embedded URL credentials, and blocks private, loopback, link-local, reserved, and other special-purpose destinations by default. Optional RSS and Atom discovery stays inside the same origin, request, depth, byte, and duration ceilings.

  • Keep private-network crawling disabled for untrusted input.
  • The trusted private-network opt-in does not permit link-local metadata endpoints.
  • Respect robots.txt, access controls, site terms, authorization, per-origin delay, response byte limits, and total crawl deadlines.
  • The offline feed parser rejects DTD and entity declarations, sanitizes embedded content, and returns only credential-free HTTP or HTTPS item links.
  • Use deployment egress controls even when application-level SSRF checks are enabled.

External coding agents

  • Keep Codex in a read-only sandbox for review tasks and Claude Code in plan mode with an explicit tool list.
  • Use workspace-write only for a workflow whose write effect is approval-gated.
  • Pass prompts over standard input, keep commands and arguments in trusted configuration, and avoid shell wrappers.
  • Allowlist working-directory roots and inherited environment keys. Do not pass unrelated credentials.
  • Set runtime, input, output, byte, turn, token, and spend ceilings where supported.
  • Treat provider activity as observed evidence. Validate the resulting files and run tests before later publish, send, deploy, or merge actions.

A provider can perform internal actions that Maqam sees only after they happen. Use provider permissions plus a container, virtual machine, restricted account, and network policy for untrusted code.

Evidence semantics

Evidence records preserve source metadata, excerpts, hashes, attribution, and same-run claim links. Scoped capabilities stamp the active run, task, and tool so a handler cannot choose trusted attribution fields.

Provenance is not truth A correct hash proves that recorded content has not changed relative to that digest. It does not prove the source is trustworthy or the claim is semantically correct.

Do not place credentials, personal data, private source text, or token-shaped values in evidence excerpts. Provider event streams can contain paths, commands, source fragments, and model output, so apply explicit retention and data-classification rules.

Deployment checklist

  1. Map every side-effect path and prove that it reaches the gateway.
  2. Keep host permissions narrower than the declared application policy.
  3. Authenticate users and reviewers outside agent-controlled input.
  4. Use integrity-protected persistence for approval, trace, and evidence state.
  5. Set process, filesystem, credential, network, origin, rate, and byte limits.
  6. Define idempotency and rollback for every operation that can partially succeed.
  7. Run the unit suite, adapter conformance, MGES fixtures, package audit, clean install, and deployment-specific tests.
  8. Review logs and evidence before a later publish, send, deploy, or merge step.

Report a security issue

Use the repository's private vulnerability reporting path. Do not put secrets, tokens, customer data, or working exploit payloads in a public issue.
