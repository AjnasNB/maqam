Benchmarking an agent governance boundary without fooling yourself

One number cannot answer latency, policy enforcement, security, and task quality at the same time.

Agent products often compress unrelated questions into one score. How fast is the runtime? Does policy stop a denied call? Is the whole system secure? Does the agent complete useful work? Each question needs a different fixture and a different claim.

The Maqam Governance Evaluation Suite, or MGES v1, keeps two profiles separate: a local-call performance microbenchmark and a deterministic governance-boundary conformance suite. The first detects regressions in one in-process path. The second exercises named invariants. Neither ranks models or competing products.

Start with the measurement claim

The performance objective is deliberately narrow: measure the sequential cost of Maqam's current local governed-call path under matched conditions. It excludes model inference, networks, storage, concurrency, human review, durable workflows, prompt-injection resistance, and task success.

The conformance objective is different: execute declared boundary cases and publish pass or fail evidence for the exact tested source fingerprint. A passing result is regression evidence, not a security score, certification, penetration test, or guarantee that every application path is governed.

Measurement ruleDefine the question and intended use before choosing a metric. Report the fixture, environment, uncertainty, artifacts, and limitations with the result.

What one governed call includes

const input = { value: 41, nested: { release: "0.2.x" } };
const handler = async (value) => value.value + 1;

The direct variant invokes the handler. The governed variant invokes it through ToolGateway with an allowlist PolicyEngine and tracing enabled.

  • Snapshot and validate caller context.
  • Canonicalize, detach, deep-freeze, and hash input.
  • Resolve the registered tool and effective goal.
  • Evaluate and normalize policy.
  • Enforce per-run limits.
  • Construct a scoped evidence facade.
  • Dispatch the exact detached input.
  • Append a redacted completion trace.

After each timed batch, the fixture verifies the result checksum, completed call count, and one trace record per governed operation. It cannot report an attractive number after silently skipping normal trace work.

Why batch sizes differ

The direct handler only performs an asynchronous addition, so a short direct batch would be dominated by scheduling noise. The governed path allocates snapshots and trace records, so it reaches a stable timing window with fewer operations.

Each observation divides its own elapsed time by its own operation count. Both median timed batches exceeded the project's 100 ms minimum in the recorded run.

Fresh processes and alternating order

JavaScript timings respond to runtime optimization, allocation, garbage collection, CPU scheduling, and execution order. MGES starts every observation in a fresh Node child process. Startup and module loading finish before the timed interval.

round 1: direct   -> governed
round 2: governed -> direct
round 3: direct   -> governed
...

Thirty paired rounds alternate order to reduce a simple temporal bias. Timing uses Node's monotonic process.hrtime.bigint().

The statistics and their limits

The headline is the median of 30 governed observation means. MGES also records the mean, sample standard deviation, coefficient of variation, median absolute deviation, minimum, maximum, 5th and 95th percentiles, and every raw observation.

The reported 95% interval is a deterministic percentile-bootstrap interval for the sample median using 10,000 resamples. It estimates sampling uncertainty across runs on the recorded machine. It does not describe uncertainty across processors, operating systems, Node versions, thermal states, or production workloads.

nanoseconds per operation = elapsed nanoseconds / completed operations
operations per second at median = 1,000,000,000 / median nanoseconds

The derived rate is a sequential local rate, not a concurrency or capacity measurement. MGES reports paired absolute added time and avoids promoting a governed/direct ratio because the direct baseline intentionally does almost no work.

Conformance is not latency

The governance-boundary profile executes 12 deterministic fixtures covering deny-by-default construction, denial before dispatch, fail-closed policy evaluation, safe input handling, exact approval scope, replay rejection, call limits, detached handler input, evidence isolation, atomic multi-approval consumption, attribution, and redacted denial traces.

The recorded artifact reports 12 of 12 fixtures passing. MGES does not turn that into a percentage security score because equal weighting would imply equal risk and coverage. That has not been established.

Relationship to OWASP agentic risks

The OWASP Top 10 for Agentic Applications 2026 is a peer-reviewed risk framework. MGES is neither a substitute nor an OWASP certification.

Several cases provide narrow evidence relevant to tool misuse: deny unknown calls, scope approval to an exact action, enforce a call limit, and consume approvals atomically. They do not test prompt injection, poisoned tool descriptions, provider-internal execution, operating-system containment, or network isolation.

Exact scope and one-use consumption are relevant to accidental authority reuse. Maqam v0.2.x still accepts reviewer identity as host-supplied data. Authentication, IAM, delegation, credentials, and non-repudiation remain host responsibilities.

An approval gate creates a review point, but MGES does not evaluate reviewer fatigue, interface quality, social engineering, or whether the host authenticated the reviewer.

Why this cannot rank Maqam against other tools

Maqam, a crawler, an orchestration framework, and a policy engine do not perform the same work. Comparing their default tool-call timing would reward whichever system does less.

A credible cross-product comparison needs the same payload and effect, the same policy and approval obligations, equivalent input scope and trace output, an agreed process model, public adapters, predeclared statistics, independent machines, disclosure of unstable runs, and review by maintainers of every system. MGES v1 does not provide those competitor adapters and makes no speed ranking.

A path to ecosystem evaluation

Adapter conformance

A shared protocol could verify that effectful calls enter the gateway, identifiers survive transport, exact approval scope is preserved, cancellation fails closed, and evidence identifiers propagate. Every result should name the adapter, provider, version, and unsupported properties. Not tested must remain different from pass.

End-to-end profiles

Local CLI, MCP transport, HTTP services, durable queues, crawler research, and multi-agent handoffs each need distinct fixtures and schemas because latency and failure models differ.

Adversarial evaluation

Security work should test malicious tool descriptions, prompt-injected content, forged reviewer identity, replay across restarts, poisoned evidence, bypass attempts, origin violations, persistence failure, and concurrent races. These are attack experiments, not extensions of a latency loop.

Public result registry

Signed artifacts indexed by suite version, source fingerprint, environment, and profile could separate one machine's observation from independently reproduced results.

How to quote the result responsibly

sample median: 126.334 to 128.942; 30 fresh-process observations; CV 5.572%).

A result without its objective, fixture, environment, uncertainty, limitations, and raw artifact is not a benchmark report. It is an anecdote.
