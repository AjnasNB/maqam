Your agent approval may not authorize the input that actually executes

How exact, one-use receipts close the gap between a review screen and an agent tool call.

Agent products increasingly add confirmation before an email is sent, a repository is modified, or a deployment starts. The screen looks reassuring: it names an action, a person approves it, and the agent continues.

But what did that decision authorize?

If approval is only a boolean or reusable id, the target can change after review. A later run can reuse the decision. The tool can receive a different object from the one policy inspected. The application can display a friendly summary while executing fields that were never reviewed.

Maqam implements a narrower property at ToolGateway: bind one human decision to one run, one registered tool, and one canonical input, then consume the decision when that handler executes.

This does not make the action correct, prove reviewer identity, or make a malicious tool safe. It closes one specific class of authority gaps at the application boundary.

The common boolean model

if (approval.status === "approved") {
  await tool.call(currentInput);
}

This answers one question: is the record marked approved? It does not necessarily answer whether the record belongs to this run and tool, whether the input still matches, whether the decision already authorized another call, or whether stored state came from an authenticated source.

The ordinary failure modes

Input mutation

Policy authorizes a caller-owned object. Before the handler reads it, another callback changes environment: "staging" to environment: "production".

Approval replay

A valid approval id is presented for a second execution, another run, or another payload.

Confused tool authority

A decision requested for a preview tool is accepted by a publish tool because approval is not scoped to the registered name.

Display and execution drift

The review screen summarizes a subset of fields while hidden or newly added input fields affect execution.

Partial multi-approval consumption

A call needs tool and effect approval. One is consumed before the other is found invalid, leaving state that is difficult to retry safely.

The Maqam call boundary

Maqam accepts finite, acyclic JSON values built from supported primitives, dense arrays, and plain objects. Accessors, symbol keys, sparse arrays, repeated object references, non-finite numbers, negative zero, and unsupported object types fail closed.

  1. Validate and detach the input.
  2. Ask policy whether the tool and declared effects are allowed.
  3. Create pending approval requests when policy requires them.
  4. Bind requests to run id, registered tool, and input hash.
  5. Return without invoking the handler.
  6. Let a trusted application reviewer decide.
  7. Retry the same call with the approval id.
  8. Validate every required approval and consume them atomically.
  9. Invoke the handler with the detached authorized input.

If the run, tool, or input changes, the approval does not match. If the approval was consumed, it cannot authorize another execution.

A runnable local proof

The handler below is a fake publisher, so it has no external effect.

import assert from "node:assert/strict";
import {
  ApprovalQueue,
  ApprovalRequiredError,
  PolicyEngine,
  ToolGateway
} from "maqam";

const approvals = new ApprovalQueue();
const policy = new PolicyEngine({
  allowedTools: ["publisher"],
  allowedOrigins: ["https://registry.npmjs.org"],
  approvalRequiredEffects: ["publish"]
});
const gateway = new ToolGateway({
  policyEngine: policy,
  approvalQueue: approvals
});

const executions = [];
gateway.registerTool("publisher", async (input) => {
  executions.push(input);
  return { published: input.version };
}, { effects: ["publish"] });

const input = {
  packageName: "example-package",
  version: "1.2.3",
  artifactIntegrity: "sha512:reviewed-example"
};
const context = { runId: "release_42" };

let request;
try {
  await gateway.call("publisher", input, context);
  assert.fail("The handler must wait for approval.");
} catch (error) {
  assert.ok(error instanceof ApprovalRequiredError);
  request = error.details.approvalRequests[0];
}

assert.equal(executions.length, 0);
approvals.approve(request.approvalId, {
  decidedBy: "authenticated-release-owner"
});

await assert.rejects(
  gateway.call("publisher", { ...input, version: "1.2.4" }, {
    ...context,
    approvalId: request.approvalId
  }),
  (error) => error.code === "APPROVAL_SCOPE_MISMATCH"
);

const result = await gateway.call("publisher", input, {
  ...context,
  approvalId: request.approvalId
});
assert.deepEqual(result, { published: "1.2.3" });
assert.equal(executions.length, 1);

await assert.rejects(
  gateway.call("publisher", input, {
    ...context,
    approvalId: request.approvalId
  }),
  (error) => error.code === "APPROVAL_INVALID"
);

In production, decidedBy must come from authenticated reviewer context, never from agent input, a query string, or an unsigned client payload.

Why the execution snapshot matters

Hash comparison is insufficient if the handler later receives the original mutable object.

const input = { environment: "staging" };
const call = governedCall(input);
input.environment = "production";
await call;

If policy hashed staging but the handler reads production, audit and execution disagree. Maqam validates and detaches the value, then uses that snapshot for policy, approval, and execution. It cannot guarantee that a remote service honors the fields correctly.

Effects belong in policy

A tool name can be too coarse. The same connector may read, write, and publish. Maqam handlers declare effects, and policy can require approval for selected effects. Registration may add effects or raise risk, but it cannot remove a handler's minimum effect set.

Effects remain application vocabulary. Maqam cannot inspect arbitrary code and discover its real side effects. Adapter authors must declare them accurately, and credentials must remain least privileged.

Approval and evidence answer different questions

Approval asks whether a configured action may execute. Evidence asks which recorded sources support a later claim.

The evidence ledger can record source metadata, excerpts, hashes, run and tool attribution, confidence, and same-run claim links. Scoped capabilities keep handlers from choosing trusted attribution fields or accessing the raw ledger.

Provenance is not truth verificationA hash shows that recorded content has a digest. A claim link shows that a workflow cites a record. Neither proves that the source is reliable or that the claim follows from it.

Bypass is still bypass

No application library can govern a path that does not call it. Direct access to a cloud SDK, deployment credential, general shell, or network endpoint remains outside Maqam.

  • Remove direct credentials from agent environments.
  • Expose narrow registered tools instead of general shells.
  • Use provider read-only or plan defaults.
  • Isolate high-risk workers in containers or virtual machines.
  • Enforce egress and service-side authorization separately.
  • Verify the external outcome after execution.

In-process state is not durable governance

Current Maqam approvals, traces, runtime state, and evidence live in process. Export and restore APIs let a host persist state, but serialization does not authenticate a decision or make consumption safe across processes.

A production persistence layer needs authenticated reviewer identity, integrity protection, compare-and-swap or transaction semantics, schema migration, crash recovery, retention, redaction, deletion, and audit access control.

A practical adoption sequence

  1. Identify one high-risk action with stable structured input.
  2. Remove direct access to that action from the agent.
  3. Register a narrow adapter with accurate effects.
  4. Configure deny-by-default tool and origin policy.
  5. Display every authority-bearing field to the reviewer.
  6. Supply reviewer identity from authenticated context.
  7. Retry only the exact gateway-generated request.
  8. Verify the outcome and record relevant evidence.
  9. Add mismatch, replay, mutation, bypass, and failure tests.
  10. Add durable storage only with explicit integrity and concurrency semantics.

Conclusion

"A human approved it" is incomplete unless the system can say what it was.

An exact receipt should identify execution context, bind the reviewed payload, reject drift, and resist replay. It should also be honest about what remains outside: reviewer authentication, handler behavior, provider-internal actions, host isolation, durable state, and semantic correctness.

Maqam is a focused TypeScript control boundary for applications that need a stronger connection between policy, approval, execution, and evidence.
